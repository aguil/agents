# ADR 0009: spec v0.2 `applies_to` hook scoping; policy-bridge cost accepted with measurement

**Status:** Accepted **Context:** Two deferred findings from the Phase 1
self-review (#69) came due as prerequisites for replacing the code-review
subcommand with a config-driven harness (#73 Tier 4): `pre_tool_call` hooks
registering on MCP events despite shell-only matchers (#71), and the policy-eval
bridge paying CLI startup plus policy parse on every hook invocation (#70).

**Decision:**

1. **Harness spec v0.2 opens with one additive field: per-handler
   `applies_to`.** Tool-call hook handlers may declare
   `applies_to: [shell | mcp | edit]`; generation only registers the handler on
   adapter events belonging to those classes. This is the "honest fix" from
   #71's option B: the canonical `pre/post_tool_call` vocabulary is coarser than
   what authors mean, and author intent is recorded explicitly rather than
   inferred by interpreting matcher regexes against an open-ended MCP tool-name
   space (option C was rejected — silent under-registration of a policy-adjacent
   hook is the wrong failure mode for a governance layer).
   - Absence means "all classes" — an unscoped handler keeps register-everywhere
     behavior, so narrowing is always opt-in.
   - `applies_to` is only valid on tool-call events; the loader rejects it
     elsewhere, along with unknown classes and empty lists.
   - The builtin policy bridge is never narrowed: it stays first on every mapped
     tool event.
   - v0.2 is additive; v0.1 documents load unchanged
     (`SUPPORTED_SPEC_VERSIONS = ["0.1", "0.2"]`).

2. **The per-hook subprocess cost of the policy bridge is accepted, with numbers
   (#70 option A).** Measured 2026-07-18 (i5-1240P, Bun 1.3.13, 30 iterations):
   p50 48.5ms per `bun run … policy-eval` invocation. Across 24 recorded
   code-review runs the median count of hook-mapped tool events (shell/MCP/edit
   — read and search calls do not map to hooks, and code-review roles are
   read-only) is 6.5 per run, projecting **~0.32s of added wall-clock per
   code-review run** — noise against multi-minute run durations and
   per-tool-call LLM latency. Full report: `docs/perf/policy-eval-hook-cost.md`.
   - Option D (compiled minimal bridge) is additionally blocked today: the full
     CLI eagerly resolves `docs/skills/skills.json` at import time and a
     `bun build --compile` binary (97.5 MiB) fails at startup. Revisit only if a
     future harness profile shows materially higher hook-mapped event counts; a
     purpose-built minimal entry point (not the full CLI) would be the shape of
     that fix.

3. **`agents hooks test` is the probe surface for policy behavior.** Feeds a
   synthetic hook event through the same normalization + evaluation path the
   runtime bridge uses and reports the verdict. Scriptable exit codes (0
   allow/warn/transform, 2 deny, 3 escalate, 1 usage/load error) make it the
   building block for the #73 Tier 4 policy-parity probe suite. Unlike
   `policy-eval`, load errors fail loudly (exit 1) instead of failing closed —
   it is a development tool, not an enforcement path.

**Consequences:**

- #71 and #70 close; #73 Tier 4's "per-hook cold-start cost measured against a
  budget" gate is satisfied by the report above.
- Spec v0.2 is the vehicle for further deferred spec extensions when they come
  due (ADR 0007 consequences list).
- The eager skills-pack resolution in the CLI entry point is a latent packaging
  constraint (blocks any compiled-binary distribution), tracked separately if
  compiled distribution becomes a goal.
