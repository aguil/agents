# Release checklist

Releases of **`@aguil/agents`** are automated with
[release-please](https://github.com/googleapis/release-please). Publishing
mechanics (npm tarball, Trusted Publishing) live in [`BUILD.md`](../BUILD.md).

## How a release ships

1. Land [Conventional Commits](https://www.conventionalcommits.org/) on
   **`main`** (`feat:` → minor, `fix:`/`perf:` → patch while pre-1.0 per
   [`release-please-config.json`](../release-please-config.json)).
2. [`.github/workflows/release-please.yml`](../.github/workflows/release-please.yml)
   maintains a **release PR** that bumps the root `package.json` version,
   updates [`CHANGELOG.md`](../CHANGELOG.md), and updates
   [`.release-please-manifest.json`](../.release-please-manifest.json).
3. Merging the release PR creates the **`vX.Y.Z`** tag and the GitHub Release
   (notes come from the changelog entry), then dispatches
   [`release.yml`](../.github/workflows/release.yml) — tags created with
   `GITHUB_TOKEN` do not trigger workflows on their own.
4. `release.yml` refuses non-tag refs and non-SemVer tag names, runs the quality
   gates (`typecheck`, `lint`, `test:ci`, `build`), packs the tarball with the
   tag's version, and publishes to npm via Trusted Publishing. The publish step
   skips when that version is already on npm, so retries are safe.

Day to day there is nothing to do beyond writing conventional commit messages
and merging the release PR when you want to ship.

## Before merging a release PR

- [ ] CI is green on **`main`** and on the release PR.
- [ ] The release PR's version bump and changelog entry look right (fix commit
      types on `main` and let release-please regenerate if not).
- [ ] Commits reaching GitHub are signed (see
      [`.agents/rules/pre-commit-checks.md`](../.agents/rules/pre-commit-checks.md)
      for the jj `git.sign-on-push` setup).

## After merging

- [ ] Tag **`vX.Y.Z`** and the GitHub Release exist.
- [ ] The **Release** workflow ran for the tag and finished green.
- [ ] [`@aguil/agents`](https://www.npmjs.com/package/@aguil/agents) shows the
      new version.
- [ ] Smoke install (optional): `npm install -g @aguil/agents@<semver>` then
      `agents --help`.

## Where release notes appear

| Surface             | What users see                                                              |
| ------------------- | --------------------------------------------------------------------------- |
| **CHANGELOG.md**    | Committed entry generated by release-please from conventional commits.      |
| **GitHub Releases** | The same changelog entry, posted by release-please when the tag is created. |
| **npm**             | `README.npm.md` + `package.json` `description` from the publish pack.       |

Do not commit per-version release notes outside `CHANGELOG.md`; the
release-please-maintained changelog is the only committed source of release
notes.

## One-time npm / CI prerequisites

Skip if already done for **`@aguil/agents`**:

- [ ] Package exists on npm (placeholder **`0.0.0`** bootstrap if needed).
- [ ] **Trusted Publishing** on npm points at this repo, workflow file
      **`release.yml`**. See [`BUILD.md`](../BUILD.md).

## Retries and recovery

- **Publish failed after the tag exists:** fix forward on `main` if the failure
  is in the code or workflow, or re-run the workflow if it was transient:
  `gh workflow run release.yml --ref vX.Y.Z`. The workflow refuses branch refs,
  and the publish step is idempotent (skips versions already on npm).
- **Do not move an already-pushed `v*.*.*` tag.** Land the fix and merge the
  next release PR instead.
- **Local tarball dry run** (optional sanity check):
  `bun run publish:npm:pack:test`.
